最近八天做一个项目，叫 zcode-wallpaper，给一个 Electron 桌面客户端（ZCode）做定制。它有一条铁律：**不改 app.asar**。app.asar 是 Electron 应用打包后的核心程序文件，改了它意味着每次官方升级都会把你的改动覆盖掉。所以整套工具集的设计哲学就是从外部下手——注入、监听、共存，而不是侵入。

这条铁律逼出了一个意外的副产品：一套被真机反复验证过的工程方法论。八天里演化出九种能力，每加一种几乎都踩一次坑，每踩一次都沉淀一条教训，到现在攒了二十八条。回头看，这些功能本身倒不算什么，真正值钱的是那些被现实按在地上摩擦出来的规则。

## 九种能力，三个层次

按"改 ZCode 的哪一面"，九种能力可以分成三层。

**改渲染层**走的是 Chrome DevTools Protocol（CDP）。ZCode 启动时带上调试端口，我就能通过 CDP 往它的主页面里塞东西：图片壁纸塞一个 style 标签设背景图，视频壁纸因为 CSS 背景根本播不了视频，所以塞一个真实的 video 元素沉到所有 UI 底下。后来又加了视频默认有声、实时切静音、还有个壁纸轮播每隔几分钟自动换图。

**改原生窗口层**完全不走 CDP。用的是 Win32 的 SetLayeredWindowAttributes，把 ZCode 主窗口本身设成半透明，能透过窗口看到后面的桌面。整个窗口均匀变淡，代码、菜单、背景一起按同一比例透明——没有"背景透明字清晰"这种美事，Win32 不提供。

**独立子应用**既不注入主页面也不改窗口。小说阅读器是一个本地 HTTP server 加前端网页，在 ZCode 自带的浏览器面板里打开；控制中心是个统一操作台，把前面所有功能收进一个透明的浮动面板；书签管理在控制中心里维护常用网址点一下就跳；还有个修复浏览器面板里 target="_blank" 链接打不开的小功能。

## 价值不在功能，在教训

这九种功能里，每一种背后都有故事。最典型的几个主题反复出现。

**先验假设错了，全白干。** 这个坑踩了至少五次。最早的一次：用户报告启动后看不到壁纸。我花了大量精力查注入脚本内部的 CDP 连接、重试逻辑、DOM 验证，甚至加了一整套重试加验证机制，测试全绿。但问题根本不在注入脚本——是启动脚本里 PowerShell 的一个 Count 陷阱，单对象时 Count 返回 null，导致注入脚本从头到尾没被调用过。

最讽刺的是，最明显的信号一直在眼前：用户每次贴的输出都停在第三步之后，第四步从没出现过。这本该第一时间让人怀疑注入根本没跑。但人就是会默认"我已经处理过了"然后往深处钻。

同样的剧本在调透明度时又来一次。我按 CSS 常识判断 rgba 的 alpha 就是控制壁纸可见度的旋钮，从零点八二调到零点九五再到一点零。调到数学上完全不透明了，壁纸还是清晰可见——这违背了 alpha 的定义。事实和理论打架的时候，一定是模型错了不是事实错了。真相是壁纸挂在 body 上，而那些吃透明变量的 UI 面板根本盖不满整个窗口，面板之间的间隙永远是满强度壁纸。

用户说"改了没效果、而且越改越没效果"，这个时候立刻要怀疑你改的参数根本不控制那个现象。不要继续在同一参数上加大力度，那是赌徒加注。

**复用而非拷贝。** 第一次只修了启动脚本，漏了另一个脚本的同一段探测逻辑，用户一直开着单窗口正好触发陷阱，又坏了。最后的解法是把探测逻辑抽成共享的文件，两个脚本都调用它。根除重复才是防再次发生的办法——两份拷贝就是两份能各自再坏一次的机会。

这个原则后来贯穿整个项目。视频壁纸复用图片的同一个注入脚本只是模式不同；控制中心只负责触发，不重写任何子系统的动作逻辑；新加的夜间模式复用 webview 注入的成熟通路。

**跨进程胶水必须真机端到端验。** 单测只能覆盖单一语言内的纯函数。C# 拼字符串然后 PowerShell 解析这种跨语言边界，两边各自合法，合起来却错位，只有真跑才抓得到。透明脚本就踩过这个坑：语法零错误，单测全绿，真机一跑却报找不到窗口。根因是 C# 和 PowerShell 两边的字段索引差了一位。

还有 Windows 上杀子进程不投递 SIGTERM 信号，直接 TerminateProcess。任何期望子进程优雅退出跑清理逻辑的设计在 Windows 上必坏。这个 bug 也是真机跑出来、单测完全抓不到的。Linux 上习以为常的信号语义，跨 OS 就不成立。

**探测脚本优先于猜测。** 遇到"为什么这个 CSS 不生效"、"元素能不能滚动"、"webview 能不能透壁纸"这类问题，第一反应不该是用 CSS 常识推理，而是写个探测脚本读真实的 computed style。

webview 能不能透壁纸这个事，讨论的时候有人断言"必盖壁纸"有人说"能透"，两个都是猜。一跑探测脚本发现 webview 元素及整条祖先链全是透明，是页面自己画实色背景盖住。结论瞬间清晰，不用再争论。

目录滚不到当前章，连改三版都失败。前两版用 scrollIntoView 和 offsetParent 链累加都不行，最后用 getBoundingClientRect 的视口坐标差值才对——因为 offsetParent 链可能通向 body 而不是你以为的容器。脑子里推演 DOM 关系十遍，不如一个脚本读一次真实状态。

**批量样本测正则。** 章节识别正则最初只拿两三本书调，自以为覆盖全了。用户丢来八十六本起点完结小说批量跑，立刻暴露十二本异常。调到第二版降到四本，靠的就是批量脚本快速反馈。

还有个反直觉的教训：加防误匹配的守卫常常得不偿失。想加守卫防正文里"他翻开第一章"这种被误识别为目录，结果误杀了一堆合法前缀，异常从四本飙到十九本。撤掉守卫回到"接受少量假阳性"反而更好。两害相权取其轻，守卫的代价要看它能误伤多少合法情况，不能只盯着它挡掉的假阳性。

## 工作流本身也是产物

每个新能力都走同一套流程：先 brainstorm 探索意图和需求，然后写设计稿把方案和边界定死，再写实施计划拆成可跟踪的任务，最后才动手。文档全放在固定目录下，比代码还详尽。

改动有明确的纪律。bash 脚本保持纯 ASCII 加 Windows 换行；PowerShell 脚本有中文必须存 UTF-8 带 BOM，否则中文 Windows 会按本地编码解析导致整文件崩；PowerShell 一律用文件执行不内联，因为某些 shell 环境会吞掉 PowerShell 里的变量。

这些看起来是琐事，但每一条背后都是一次血泪事故。存错换行符，脚本双击直接崩；漏了 BOM，here-string 解析全乱；变量被吞，报一堆看不懂的错。

## 现在的状态

八天后项目很健康。二十四测试全绿，工作区干净，文档完备。下一步是在已有的 webview 注入通路上挂一个夜间模式，架构完全对称已有的模块。虽然宽覆盖式深黑 CSS 对纯文字站效果好、对复杂图文站可能显示异常是简化版的固有代价，但能用最小工程量覆盖最常见的夜间阅读场景。

回头看，最大的收获不是堆了多少功能。功能谁都能写，AI 帮你写更快。真正稀缺的是那套方法论：先查组件边界再钻内部、探测脚本优先于猜测、跨进程胶水必真机验、复用而非拷贝、批量样本测正则。

这些规则没有一条是从文档里读出来的，全是被现实按在地上摩擦出来的。每一条都对应一次白干，每一次白干都对应一个本该看见却忽略的信号。方法论的价值在于，下一个项目起步时，不用再把每个坑重新踩一遍。
